## 성과발표회 자료

## 1. 추진배경

### 요약 1: PM 업무의 비효율성 문제

프로젝트 관리자(PM)는 개발 프로젝트의 진행 상황을 파악하고 팀원들과 소통하며 상위 관리자에게 보고하는 업무를 수행합니다. 
그러나 이러한 업무는 대부분 수동으로 이루어지고 매우 느립니다. PM은 개발자들에게 직접 질문하거나, GitHub 저장소를 수동으로 확인하거나, 프로젝트 관리 도구와 소스코드 저장소 간의 정보를 별도로 수집하여 비교 분석해야 합니다. 이는 반복적이고 시간 소모적인 작업이며, 실시간으로 정확한 진행 상황을 파악하기 어렵습니다.

**상세 설명:**

현재 대부분의 소프트웨어 개발 프로젝트에서는 프로젝트 관리 도구(예: Jira, Trello)와 소스코드 저장소(예: GitHub)가 분리되어 운영됩니다. PM은 다음과 같은 문제에 직면합니다:

1. **수동 진행도 파악의 한계**: PM은 개발자에게 직접 질문하거나 주기적인 미팅을 통해 진행 상황을 파악해야 합니다. 이는 실시간성이 떨어지고, 개발자의 주관적 판단에 의존할 수밖에 없습니다.

2. **소스코드와 관리 도구 간 단절**: GitHub에 실제로 커밋된 코드와 프로젝트 관리 도구의 Task 상태가 일치하지 않는 경우가 빈번합니다. 개발자가 코드를 작성했지만 Task 상태를 업데이트하지 않았거나, 반대로 Task는 완료로 표시되었지만 실제 코드 변경이 없는 경우가 발생합니다.

3. **반복적인 보고 업무**: 상위 관리자나 고객에게 진행 상황을 보고할 때마다 PM은 여러 소스에서 정보를 수집하고, 이를 정리하여 보고서를 작성해야 합니다. 이는 주기적으로 반복되는 업무이며, 보고 시점에 따라 정보의 일관성이 떨어질 수 있습니다.

4. **팀원별 기여도 파악의 어려움**: 여러 프로젝트를 동시에 진행하거나 팀원이 많을 경우, 각 팀원의 기여도를 정확히 파악하기 어렵습니다. 커밋 수, 코드 변경량, 이슈 해결 수 등을 수동으로 집계해야 합니다.

### 요약 2: 자동화를 통한 효율성 개선 필요성

이러한 비효율성을 해결하기 위해서는 GitHub 저장소와 프로젝트 관리 도구를 자동으로 연동하고, 소스코드 기반으로 진행도를 자동 분석하며, AI를 활용하여 누락된 작업을 제안하는 자동화 시스템이 필요합니다. 이를 통해 PM은 수동 작업에 소요되던 시간을 절약하고, 더 정확하고 실시간적인 프로젝트 진행 상황을 파악할 수 있으며, 전략적 의사결정에 집중할 수 있습니다.

**상세 설명:**

자동화 플랫폼을 구축함으로써 다음과 같은 개선 효과를 기대할 수 있습니다:

1. **GitHub 연동을 통한 자동 데이터 수집**: GitHub REST API를 활용하여 커밋, 이슈, 브랜치 정보를 자동으로 수집하고 데이터베이스에 저장합니다. PM은 수동으로 GitHub를 확인할 필요 없이 플랫폼에서 통합된 정보를 확인할 수 있습니다. Public 저장소는 URL만으로 접근 가능하며, Private 저장소의 경우 Personal Access Token을 입력하여 접근할 수 있습니다.

2. **AI 기반 분석 및 제안**: 소스코드 분석을 통해 누락된 작업이나 리팩토링이 필요한 영역을 AI가 자동으로 제안합니다. 커밋 히스토리와 현재 Task 목록을 분석하여 새로운 Task를 우선순위와 함께 제안함으로써, PM이 놓치기 쉬운 작업을 사전에 파악할 수 있습니다.

3. **실시간 진행도 추적**: Task 완료율과 코드 기반 진행도를 자동으로 계산하여 실시간으로 표시합니다. 커밋 수, 라인 변경량, 브랜치 수, 이슈 완료율 등을 종합적으로 분석하여 프로젝트의 실제 진행 상황을 정확히 파악할 수 있습니다.

4. **Task-커밋 자동 연동**: Task와 GitHub 이슈를 연결하면, 이슈가 닫히면 자동으로 Task 상태가 완료로 업데이트됩니다. 또한 커밋 메시지에 이슈 번호가 포함되면 자동으로 연동되어, 개발자의 수동 작업을 최소화합니다.

5. **팀원별 기여도 자동 집계**: 각 팀원의 커밋 수, 이슈 할당/완료 수, 코드 변경량을 자동으로 집계하여 기여도를 시각화합니다. 이를 통해 공정한 평가와 업무 분배가 가능합니다.

이러한 자동화를 통해 PM은 반복적인 수집 및 정리 업무에서 벗어나, 프로젝트 전략 수립, 리스크 관리, 팀원 관리 등 고부가가치 업무에 집중할 수 있게 됩니다.

## 3. 주요기능 설명

### 3.1 GitHub 연동 및 자동 동기화

본 플랫폼의 핵심 기능 중 하나는 GitHub 저장소와의 자동 연동입니다. PM은 프로젝트 생성 시 또는 이후에 GitHub 저장소 URL을 입력하기만 하면, 플랫폼이 자동으로 커밋, 이슈, 브랜치 정보를 수집합니다.

**주요 특징:**
- **Public/Private 저장소 지원**: Public 저장소는 URL만으로 접근 가능하며, Private 저장소의 경우 Personal Access Token을 입력하여 접근할 수 있습니다.
- **프로젝트별 토큰 관리**: 각 프로젝트마다 독립적으로 GitHub 토큰을 저장하여, 여러 프로젝트를 동시에 관리할 수 있습니다.
- **자동 동기화**: 수동 동기화 버튼을 통해 최신 커밋, 이슈, 브랜치 정보를 가져옵니다. 커밋의 경우 기본 정보뿐만 아니라 추가/삭제된 라인 수, 변경된 파일 수 등 상세 통계 정보도 함께 수집합니다.
- **데이터베이스 저장**: 수집한 정보는 데이터베이스에 저장되어, GitHub API Rate Limit에 영향받지 않고 빠르게 조회할 수 있습니다.

**동작 흐름:**
1. 프로젝트 소유자가 GitHub 저장소 URL 입력
2. Private 저장소인 경우 Personal Access Token 입력 (선택사항)
3. "동기화" 버튼 클릭
4. 백엔드가 GitHub API를 호출하여 커밋/이슈/브랜치 정보 수집
5. 데이터베이스에 저장 (중복 방지)
6. 프론트엔드에서 저장된 데이터 조회 및 표시

### 3.2 소스코드 기반 진행도 분석

기존 프로젝트 관리 도구는 Task 완료율만을 기준으로 진행도를 계산합니다. 그러나 실제로는 코드가 작성되지 않았거나, 반대로 코드는 작성되었지만 Task가 완료로 표시되지 않은 경우가 발생합니다. 본 플랫폼은 Task 기반 진행도와 코드 기반 진행도를 별도로 계산하여, 더 정확한 진행 상황을 파악할 수 있도록 합니다.

**진행도 계산 방식:**
- **Task 기반 진행도**: 완료된 Task 수 / 전체 Task 수
- **코드 기반 진행도**: 다음 지표들을 종합하여 계산
  - 커밋 수 및 최근 커밋 활동
  - 코드 변경량 (추가/삭제된 라인 수)
  - 브랜치 수 및 브랜치 활동
  - 이슈 완료율 (닫힌 이슈 수 / 전체 이슈 수)

**팀원별 기여도 분석:**
- 각 팀원의 커밋 수
- 이슈 할당 및 완료 수
- 코드 변경량 (라인 추가/삭제)
- 최근 활동 기간

이러한 정보를 종합하여 프로젝트의 실제 진행 상황을 정확히 파악하고, 팀원별 기여도를 객관적으로 평가할 수 있습니다.

### 3.3 AI 기반 Task 제안

AI 기능은 본 플랫폼의 차별화된 기능입니다. 소스코드와 커밋 히스토리를 분석하여, 누락된 작업이나 개선이 필요한 영역을 자동으로 제안합니다.

**AI 분석 프로세스:**
1. **커밋 히스토리 분석**: 최근 커밋 메시지와 변경 사항을 분석하여 패턴 파악
2. **현재 Task 목록 분석**: 이미 등록된 Task와 비교하여 누락된 작업 식별
3. **코드 품질 분석**: 중복 코드, 리팩토링 필요 영역 식별
4. **Task 제안 생성**: 제목, 설명, 우선순위(High/Medium/Low), 예상 소요 시간을 포함한 Task 제안 목록 생성

**멀티 에이전트 시스템:**
- **의도 분류**: 사용자의 질문이나 요청을 분석하여 적절한 Agent 선택
- **Agent 유형**:
  - Task 제안 Agent: 새로운 Task 제안
  - 진행도 분석 Agent: 프로젝트 진행도 분석 및 리포트 생성
  - Task 완료 확인 Agent: Task 완료 여부 판단
  - Task 할당 Agent: Task를 적절한 팀원에게 할당 제안
  - 일반 Q&A Agent: 프로젝트 관련 질문 답변

**다단계 분석 프로세스:**
- 최대 10단계까지 반복 분석을 통해 정확도 향상
- 각 단계에서 정보 충분성 평가
- 부족한 정보가 있으면 추가 데이터 수집 (예: 특정 파일 내용 조회)

### 3.4 Task-이슈 자동 연동

Task와 GitHub 이슈를 연결하면, 이슈 상태 변경 시 Task 상태가 자동으로 업데이트됩니다. 이를 통해 개발자는 Task 상태를 수동으로 업데이트할 필요 없이, GitHub에서 이슈만 관리하면 됩니다.

**연동 방식:**
- Task 생성 시 GitHub 이슈 번호 입력
- GitHub 동기화 시 이슈 상태 확인
- 이슈가 "closed" 상태가 되면 연결된 Task 상태를 "done"으로 자동 업데이트
- 커밋 메시지에 이슈 번호가 포함되면 자동으로 Task와 연동

이를 통해 프로젝트 관리 도구와 소스코드 저장소 간의 정보 일관성을 유지하고, 개발자의 업무 부담을 줄일 수 있습니다.

## 4. 개발 특이사항

### 4.1 멀티 에이전트 시스템

본 플랫폼의 AI 기능은 단순한 챗봇이 아닌, 사용자의 의도를 분석하여 적절한 전문 Agent를 선택하는 멀티 에이전트 시스템으로 구현되었습니다.

**의도 분류 프로세스:**
1. 사용자 메시지와 대화 히스토리를 분석
2. 프로젝트 컨텍스트(커밋, 이슈, Task 정보)를 고려
3. LLM을 활용하여 의도 분류
4. 적절한 Agent 선택 (task_suggestion_agent, progress_analysis_agent, task_completion_agent, task_assignment_agent, general_qa_agent 등)

**Agent별 특화 기능:**
- 각 Agent는 해당 도메인에 특화된 프롬프트와 분석 로직을 가짐
- Agent 간 독립적으로 동작하며, 필요시 다른 Agent를 호출할 수 있음
- 각 Agent는 다단계 분석을 지원하여 정확도 향상

### 4.2 다단계 분석 프로세스

AI Agent는 한 번의 분석으로 결과를 도출하는 것이 아니라, 최대 10단계까지 반복 분석을 수행합니다.

**분석 단계:**
1. **초기 분석**: 사용자 요청과 컨텍스트를 기반으로 초기 분석 수행
2. **정보 충분성 평가**: 현재 수집된 정보로 충분한 답변을 할 수 있는지 평가
3. **추가 정보 수집**: 부족한 정보가 있으면 GitHub API를 통해 추가 데이터 수집 (예: 특정 파일 내용, 커밋 상세 정보)
4. **후속 분석**: 추가 정보를 바탕으로 더 정확한 분석 수행
5. **최종 결과 생성**: 충분한 정보가 수집되면 최종 결과 생성

이러한 다단계 분석을 통해 단순한 키워드 매칭이 아닌, 실제 코드와 프로젝트 상황을 깊이 있게 이해한 분석 결과를 제공할 수 있습니다.

### 4.3 로컬 LLM 및 클라우드 LLM 지원

AI 백엔드는 로컬 LLM(Ollama)과 클라우드 LLM(OpenAI)을 모두 지원합니다.

**로컬 LLM (Ollama):**
- qwen2.5:14b 모델 사용 (정확도 우선)
- GPU 가속 지원
- API 비용 없음
- 데이터 프라이버시 보장

**클라우드 LLM (OpenAI):**
- GPT 모델 사용
- 높은 성능 및 안정성
- API 비용 발생
- 환경 변수로 선택 가능

**장점:**
- 개발 및 테스트 단계에서는 로컬 LLM 사용으로 비용 절감
- 프로덕션 환경에서는 필요에 따라 클라우드 LLM 사용 가능
- 사용자의 요구사항과 예산에 따라 유연하게 선택 가능

### 4.4 프로젝트별 GitHub 토큰 관리

각 프로젝트마다 독립적으로 GitHub Personal Access Token을 저장하고 관리합니다.

**특징:**
- 프로젝트 소유자만 토큰 입력/수정 가능
- 토큰은 데이터베이스에 암호화되어 저장
- API 응답에는 토큰 값이 포함되지 않음 (보안)
- 토큰 업데이트 시 빈 문자열 입력 시 기존 토큰 유지 기능
- 프로젝트의 모든 멤버가 같은 토큰을 공유하여 사용

**장점:**
- 여러 프로젝트를 동시에 관리할 때 각각 다른 토큰 사용 가능
- Private 저장소 접근 시 높은 Rate Limit (5,000회/시간) 활용
- 토큰 관리의 유연성 및 보안성 확보

### 4.5 프롬프트 최적화

AI Agent의 성능을 향상시키기 위해 프롬프트 엔지니어링을 수행하였습니다.

**최적화 전략:**
- 각 Agent 유형별로 특화된 프롬프트 템플릿 작성
- 초기 프롬프트와 후속 프롬프트를 분리하여 컨텍스트 관리
- 단계별 프롬프트를 통해 점진적 분석 수행
- JSON 형식 응답을 강제하여 파싱 오류 방지

이를 통해 AI의 응답 품질과 일관성을 향상시켰습니다.

## 5. 문제점 및 향후 개선 방안

### 5.1 현재 미완성 기능

**진행도 시각화 미구현:**
- 백엔드 API는 구현되었으나, 프론트엔드에서 진행도를 시각화하는 컴포넌트가 미완성 상태입니다.
- **개선 방안**: Chart.js 또는 Recharts를 활용한 진행도 차트 컴포넌트 구현, 프로젝트 상세 페이지에 "진행도" 탭 추가

**AI 기능 프론트엔드 연동 미완성:**
- AI 백엔드는 구현되었으나, 프론트엔드에서 AI API를 호출하고 결과를 표시하는 기능이 미완성입니다.
- **개선 방안**: AI API 함수 추가, AI 어드바이저 페이지 기능 구현, 제안된 Task를 실제 Task로 생성하는 기능 추가

**Task-이슈 자동 연동 미완성:**
- Task에 GitHub 이슈 번호를 연결하는 기능은 있으나, 이슈 상태 변경 시 Task 자동 업데이트 기능이 미완성입니다.
- **개선 방안**: GitHub 동기화 시 이슈 상태 확인 로직 추가, 이슈가 닫히면 연결된 Task 상태를 "done"으로 자동 업데이트

### 5.2 기술적 제약사항

**GitHub API Rate Limit:**
- Public 저장소: 인증 없이 60 requests/hour (IP 기준)
- Private 저장소: 인증 시 5,000 requests/hour (토큰 소유자 기준)
- **개선 방안**: 
  - 효율적인 캐싱 전략 구현
  - 동기화 주기 조절
  - Webhook을 활용한 실시간 업데이트 (향후)

**AI 응답 시간:**
- 로컬 LLM 사용 시 응답 시간이 길 수 있음 (특히 14b 모델)
- **개선 방안**:
  - 더 작은 모델 사용 (7b, 3b) 옵션 제공
  - 비동기 처리로 사용자 경험 개선
  - 응답 스트리밍 구현

**데이터베이스 성능:**
- 현재 SQLite 사용 (개발 환경)
- 대용량 데이터 처리 시 성능 이슈 가능
- **개선 방안**: 프로덕션 환경에서는 MySQL 또는 PostgreSQL로 마이그레이션

### 5.3 향후 개선 계획

**단기 개선 (1-2주):**
1. 진행도 시각화 컴포넌트 구현
2. AI 기능 프론트엔드 연동 완성
3. Task-이슈 자동 연동 기능 완성
4. 버그 수정 및 테스트

**중기 개선 (1-2개월):**
1. GitHub Webhook을 활용한 실시간 동기화
2. 진행도 예측 분석 (머신러닝 기반)
3. 코드 리뷰 기반 Task 완료 확인
4. 리팩토링 제안 기능 고도화
5. 모바일 앱 최적화 (Morpheus React 활용)

**장기 개선 (3-6개월):**
1. 다중 저장소 지원 (GitLab, Bitbucket 등)
2. 프로젝트 템플릿 기능
3. 대시보드 커스터마이징
4. 알림 시스템 (이메일, 슬랙 등)
5. 리포트 자동 생성 및 스케줄링
6. 팀 간 프로젝트 공유 및 협업 기능 강화

## 6. 활용효과 및 개선방안

### 6.1 PM 업무 효율화 효과

**시간 절약:**
- 수동 진행도 파악 시간: 주 2-3시간 → 자동화로 거의 0시간
- 보고서 작성 시간: 회당 1-2시간 → 자동 리포트 생성으로 10분 이내
- GitHub 정보 수집 시간: 회당 30분-1시간 → 자동 동기화로 즉시 확인

**정확도 향상:**
- 수동 수집 시 발생하는 오류 및 누락 방지
- 실시간 데이터 기반으로 항상 최신 정보 제공
- 코드 기반 진행도와 Task 기반 진행도를 비교하여 더 정확한 상황 파악

**전략적 업무 집중:**
- 반복적인 수집 및 정리 업무에서 벗어나, 프로젝트 전략 수립, 리스크 관리, 팀원 관리 등 고부가가치 업무에 집중 가능

### 6.2 팀 협업 개선

**투명성 향상:**
- 모든 팀원이 동일한 정보를 실시간으로 확인 가능
- 팀원별 기여도가 객관적으로 표시되어 공정한 평가 가능
- 프로젝트 진행 상황이 시각적으로 명확하게 표시

**소통 효율성:**
- 개발자는 GitHub에서만 작업하면 Task 상태가 자동 업데이트
- PM은 별도 질문 없이 플랫폼에서 진행 상황 확인
- 불필요한 미팅 및 보고 시간 감소

**책임 명확화:**
- Task 할당 및 담당자가 명확히 표시
- 각 Task의 진행 상황과 관련 커밋/이슈가 자동으로 연동
- 프로젝트 멤버 간 역할과 책임이 명확해짐

### 6.3 확장 가능성

**기능 확장:**
- 현재는 GitHub만 지원하나, GitLab, Bitbucket 등 다른 저장소도 추가 가능
- 다양한 프로젝트 관리 도구와의 연동 가능 (Jira, Trello 등)
- AI 기능의 지속적 개선 및 새로운 Agent 추가 가능

**규모 확장:**
- 소규모 팀(5-10명)부터 중규모 팀(50-100명)까지 지원 가능
- 다중 프로젝트 동시 관리 지원
- 조직 단위 관리 기능 추가 가능

**산업 적용:**
- 소프트웨어 개발 프로젝트뿐만 아니라, 다른 종류의 프로젝트에도 적용 가능
- 교육 기관의 프로젝트 관리, 오픈소스 프로젝트 관리 등 다양한 분야에 활용 가능

### 6.4 비즈니스 가치

**비용 절감:**
- PM 업무 시간 절약으로 인건비 절감
- 프로젝트 지연 방지로 인한 손실 최소화
